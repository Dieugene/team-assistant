# Concept — видение системы

---

## 1. Проблема

Системы оперативного планирования, коммуникаций и таск-трекинга
навязывают пользователям предзаданную онтологию (проекты, эпики, задачи,
стадии, доски и т.д.). Это создаёт барьер: чтобы работать с инструментом,
команда должна освоить и принять эту онтологию. На практике люди откладывают
или полностью отказываются от использования таких систем.

Проблема усугубляется с ростом сложности: чем сложнее управляемая система,
тем более сложная онтология требуется для управления ею, и тем выше
порог входа — причём для всей команды, а не только для руководителей.

При этом промежуточные стадии работы (осмысление, планирование, отчётность)
**действительно нуждаются** в структурированных онтологиях. Но конечная цель
всех этих онтологий одна: определить, какие задачи должен решать каждый
исполнитель и какой контекст ему для этого нужен.

---

## 2. Подход

Система принимает **сырой вход** (input) от пользователей в естественной форме —
без требования структурировать данные по какой-либо заранее заданной схеме.

Внутри система **преобразует** этот вход по подходящей для данного контекста
онтологии (processing) — используя AI для осмысления, классификации,
планирования и другой обработки.

На выходе система порождает **годные к использованию результаты** (output) —
задачи, контекст, уведомления — адресованные конкретным исполнителям
в понятной им форме.

```
User (естественный диалог)
  → input (сырые данные, никакой онтологии)
    → processing (AI + подходящая онтология)
      → output (готовый результат для конкретного человека)
```

Онтология не исчезает — она перемещается внутрь системы, где AI берёт
на себя бремя работы с ней. Пользователь общается естественно;
структура возникает автоматически.

---

## 3. Ключевые принципы

### Минимальный порог входа
Взаимодействие через естественный диалог. Пользователю не нужно
осваивать онтологию инструмента — достаточно говорить о своей работе.

### Онтология как внутренний механизм
Структурные модели (задачи, зависимости, приоритеты) существуют внутри
системы как инструмент Processing, а не как требование к пользователю.
Конкретная онтология может адаптироваться под контекст Team.

### AI как медиатор
AI выполняет преобразование между сырым входом и структурированным
результатом. Это включает: извлечение смысла, классификацию,
планирование, формулирование output для конкретного адресата.

### Наблюдаемость
Все процессы внутри системы прозрачны для наблюдения. Trace фиксирует
всю активность; VS UI позволяет видеть, что происходит, почему и для кого.
Это критично и для отладки, и для доверия к системе.

### Симуляция
Система должна быть тестируемой без реальных пользователей. SIM позволяет
создавать сценарии, воспроизводить ситуации и верифицировать поведение
до вовлечения реальной команды.

---

## 4. Стратегия развития

**Фаза 1: Ядро + наблюдаемость**
Базовый слой (Dialogue management, Communication, Processing, Persistence)
и слой визуализации (VS UI, Tracker). Цель — работающий прототип,
в котором видно, что происходит внутри.

**Фаза 2: Пользовательский интерфейс**
Подключение реального канала коммуникации (Telegram-бот, веб-приложение
или иное). Переход от SIM-тестирования к работе с реальными пользователями.

**Фаза 3: Адаптивные онтологии**
Расширение Processing: система не просто использует фиксированную
внутреннюю онтологию, а подбирает и адаптирует её под конкретную Team.

---

## 5. Границы MVP

### Входит в MVP
- Dialogue management: один AI-ассистент на каждого User
- EventBus с тремя Topics (input, processed, output)
- Минимум один ProcessingAgent (извлечение задач или подобное)
- OutputRouter в минимальной реализации
- Tracker, пишущий всё в Storage
- VS UI с базовыми представлениями (Timeline, диалоги)
- SIM для тестирования без реальных пользователей
- Storage на SQLite
- Масштаб: до 50 пользователей (реальных или виртуальных)

### Не входит в MVP
- Пользовательский интерфейс (Telegram, web-app) — только SIM
- Адаптивные онтологии — фиксированная внутренняя модель
- Масштабирование, отказоустойчивость, production-grade хранилище
- Связи между TraceEvents (отложено)
- Точное определение реализации OutputRouter (роль определена,
  реализация — по результатам практики)
